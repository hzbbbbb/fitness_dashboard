# DEVELOPMENT LOG RULES

## 记录目的

本文件用于规定“AI 驱动开发记录”应该如何书写。

开发记录的目标不是替代 Git 提交历史，而是补充以下信息：

- 任务背景
- 用户给 AI 的目标
- 实际完成范围
- 本轮未完成但值得继续跟进的事项

开发记录应帮助后续协作快速理解一轮任务做了什么、为什么这样做、还剩什么。

## 适用范围

适用于仓库内以下模块：

- `desktop`
- `mobile`
- `doc`

只要属于真实发生过的一轮开发任务，都应记录到开发记录目录中。

## 规范文件与真实记录的边界

- 本文件是“规范文件”，只定义写法，不记录真实事件
- 真实开发记录必须单独放在 `doc/development_logs/` 中
- 不要再把真实开发记录直接写在规范文件里

## 文件命名规范

真实开发记录文件统一放在：

```text
doc/development_logs/
```

推荐命名格式：

```text
YYYY-MM-DD-<module>-<topic>.md
```

示例：

```text
2026-03-20-mobile-apple-health-and-score.md
2026-03-20-doc-structure-cleanup.md
2026-03-21-desktop-home-layout-tuning.md
```

命名要求：

- 日期必须放在最前面
- 模块使用 `desktop`、`mobile`、`doc`
- topic 使用英文短词串联，便于文件排序和长期维护

## 每条记录必须包含的字段

每条真实开发记录至少包含以下内容：

1. 日期
2. 任务标题
3. 涉及模块
4. 任务背景
5. 我给 AI 的目标
6. 实际完成内容
7. 影响范围
8. 未完成项或后续待办

## 推荐格式

推荐使用以下 Markdown 结构：

```md
# 任务标题

## 日期

## 涉及模块

## 任务背景

## 我给 AI 的目标

## 实际完成内容

## 影响范围

## 后续待办
```

要求：

- 标题直接概括这轮任务
- “实际完成内容”只写已经落地的结果
- “后续待办”不要写空泛口号，应写清下一步可继续推进的点

## 示例模板

```md
# 移动端 Apple 健康接入与评分页首版

## 日期
2026-03-20

## 涉及模块
- mobile

## 任务背景
需要在移动端接入 Apple 健康数据，并补齐首页健康摘要与评分页展示。

## 我给 AI 的目标
- 接入 HealthKit 只读权限
- 打通步数和睡眠读取
- 修正睡眠统计口径
- 新增健康评分页面第一版

## 实际完成内容
- 新增 Apple 健康只读接入
- 首页显示今日步数与昨晚睡眠时长
- 修正授权状态误判
- 新增底部导航“评分”页

## 影响范围
- 移动端首页
- 移动端评分页
- iOS HealthKit 读取链路

## 后续待办
- 真机继续校验评分展示细节
- 根据后续反馈调整评分分段与文案
```
