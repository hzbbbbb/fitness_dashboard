# CONTRIBUTION

## 适用范围

本文件是仓库级唯一提交规范，适用于：

- `desktop`
- `mobile`
- `doc`

后续提交说明统一使用中文，不再混用英文提交格式。

## 提交信息格式

统一格式：

```text
<表情> <改动项目> <改动内容>
```

格式要求：

- 表情必须放在最前面
- 表情后保留一个空格
- 改动项目只能是 `desktop`、`mobile`、`doc`
- 改动内容必须使用中文
- 改动内容必须直接说明本次提交做了什么
- 整条提交信息应简洁明确，避免空泛描述

## 合法示例

```text
✨ mobile 新增 Apple 健康步数读取
🩹 mobile 修复睡眠时长统计错误
🎨 desktop 调整首页卡片间距
📝 doc 更新提交规范
🔧 mobile 调整 iOS 签名配置
♻️ desktop 重构记录页筛选逻辑
🔥 doc 删除子项目旧文档目录
🚀 desktop 优化图表加载性能
✅ mobile 补充健康摘要显示校验
```

## 推荐表情

常用表情建议如下：

- `✨` 新增功能
- `🩹` 修复问题
- `🎨` 样式或界面调整
- `📝` 文档更新
- `🔧` 配置修改
- `♻️` 重构整理
- `🔥` 删除废弃内容
- `🚀` 性能优化
- `✅` 测试补充

只要语义准确，也可以使用其他清晰、稳定的表情，但不要滥用。

## 不推荐写法

以下写法不符合规范：

```text
update
fix bug
✨ 更新一下
📝 doc change rules
feat(stage1): add healthkit
✨ mobile add health data
🐛 修复一些问题
```

不推荐原因：

- 没有按 `表情 + 改动项目 + 改动内容` 组织
- 改动内容不是中文
- 描述过于模糊，无法快速判断提交目的
- 混入旧的 `feat/fix/docs(stage)` 风格

## 提交粒度建议

- 一次提交只解决一个清晰问题
- 功能新增、问题修复、样式调整、文档整理尽量分开提交
- 不要把多个无关模块改动塞进同一条提交
- 如果同时改了 `desktop` 和 `mobile`，优先拆成两次提交
- 如果只是整理文档，就只提交 `doc` 相关内容

推荐拆分示例：

```text
✨ mobile 新增 Apple 健康步数读取
🩹 mobile 修复睡眠时长统计错误
📝 doc 更新 AI 协作开发日志模板
```

不推荐拆分示例：

```text
✨ mobile 新增健康数据并修复睡眠问题顺便改桌面首页和文档
```

## 提交前检查建议

提交前至少检查以下几点：

- 提交信息是否符合 `表情 + 改动项目 + 改动内容`
- 改动项目是否只使用 `desktop`、`mobile`、`doc`
- 改动内容是否为中文且足够明确
- 本次提交是否只包含当前任务相关文件
- 是否混入了无关格式化、临时文件或调试产物
- 是否存在本应拆分但还未拆分的无关改动

## AI 与人工协作要求

- AI 提交时必须遵守本文件，不得继续使用旧的 stage 提交格式
- 人工补提交时也应遵守同一规范，保持仓库历史统一
- 如果一次任务横跨多个模块，应先拆分改动，再分别提交
- 若无法明确归类到 `desktop`、`mobile`、`doc`，应先整理改动范围，不要勉强提交

## 最终规范

本仓库提交信息统一为：

```text
<表情> <改动项目> <改动内容>
```

其中：

- 改动项目只能是 `desktop`、`mobile`、`doc`
- 改动内容必须是中文
- 提交信息必须清晰、简洁、可回溯
