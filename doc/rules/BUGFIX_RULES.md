# BUGFIX RULES

## 记录目的

本文件用于规定“Bug 修复记录”应该如何书写。

Bug 修复记录的重点不是记录提交，而是保留一条问题从出现到修复的完整链路，方便后续：

- 排查同类问题
- 回顾根因
- 复用修复思路
- 明确验证方式

## 适用场景

以下情况应写入 bug 修复记录：

- 出现了真实可复现的问题
- 已经明确定位到根因
- 已经做了修复
- 即使没有完全解决，也已经形成了阶段性结论

不适用场景：

- 纯功能开发但没有实际 bug
- 仅有猜测、没有问题现象和处理过程的临时想法

## 规范文件与真实记录的边界

- 本文件是“规范文件”，只定义 bug 修复记录应该怎么写
- 真实 bug 修复记录必须单独放在 `doc/bugfix_logs/` 中
- 不要把真实 bug 过程直接混在规范文件里

## 文件命名规范

真实 bug 修复记录统一放在：

```text
doc/bugfix_logs/
```

推荐命名格式：

```text
YYYY-MM-DD-<module>-<bug-topic>.md
```

示例：

```text
2026-03-20-mobile-healthkit-authorized-but-denied.md
2026-03-20-mobile-sleep-duration-last-segment-only.md
2026-03-20-mobile-score-screen-weight-compile-error.md
```

命名要求：

- 日期放在最前面
- 模块使用 `desktop`、`mobile`、`doc`
- 主题聚焦一个具体 bug，不要混多个无关问题

## 每条 bug 修复记录必须包含的字段

每条真实 bug 修复记录至少包含：

1. 日期
2. Bug 标题
3. 涉及模块
4. 问题现象
5. 复现条件
6. 根因分析
7. 修复方法
8. 验证方式
9. 后续注意事项

## 推荐格式

推荐使用以下 Markdown 结构：

```md
# Bug 标题

## 日期

## 涉及模块

## 问题现象

## 复现条件

## 根因分析

## 修复方法

## 验证方式

## 后续注意事项
```

要求：

- “问题现象”写真实看到的结果
- “根因分析”写明确结论，不要只写猜测
- “修复方法”写实际采取的改动
- “验证方式”必须可执行

## 示例模板

```md
# HealthKit 已授权但首页仍显示无权限

## 日期
2026-03-20

## 涉及模块
- mobile

## 问题现象
用户已在系统弹窗中允许读取步数和睡眠，但首页仍显示“未获得 Apple 健康读取权限”。

## 复现条件
- 真机安装应用
- 授予 HealthKit 步数和睡眠读取权限
- 返回首页查看健康摘要

## 根因分析
应用内错误使用了不适合读取权限判断的授权状态 API，导致已授权场景被误判成 denied。

## 修复方法
- 移除错误的前置权限拦截
- 改为实际执行查询
- 仅在查询明确返回授权拒绝错误时展示无权限状态

## 验证方式
- 重新安装并授权
- 返回首页查看是否展示步数和睡眠
- 前后台切换后再次确认数据会刷新

## 后续注意事项
- 读取权限状态不要仅依赖错误的同步判断
- 优先以查询结果和实际错误码为准
```
