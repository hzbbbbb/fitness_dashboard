# CONTRIBUTING

## 提交代码规范

为了保证仓库的提交记录清晰、统一、易读，所有提交信息请遵循以下格式：

```text
<emoji> <简短描述>

基本要求
	•	提交信息开头必须使用一个表情。
	•	表情后面必须保留一个空格，再写本次改动的简短描述。
	•	描述应直接说明本次提交的核心改动内容。
	•	描述尽量简洁明确，建议控制在 20 个字以内。
	•	一次提交尽量只包含一类改动。

正确示例

✨ 新增微信登录接口
🐛 修复订单状态更新异常
♻️ 重构用户鉴权逻辑
📝 更新项目文档
🎨 调整首页按钮样式
🚀 优化列表加载性能
✅ 补充登录流程测试
🔧 修改构建配置
🔥 删除废弃代码

错误示例

update
fix bug
改了一下
提交代码
再次修改
test

这些写法的问题在于描述过于模糊，无法快速看出本次提交到底做了什么。

⸻

常用表情含义

表情	含义	使用场景
✨	新功能	新增页面、接口、模块、功能逻辑
🐛	修复问题	修复 bug、异常、错误逻辑
♻️	重构	重构代码结构，不改变核心功能
📝	文档	更新 README、接口文档、说明文档
🎨	样式/UI	调整页面布局、样式、视觉表现
🚀	性能优化	优化速度、加载性能、查询效率
✅	测试	新增或修改测试代码
🔧	配置/构建	修改配置文件、构建脚本、环境变量
🔥	删除	删除废弃代码、无用文件、旧逻辑
🚑	紧急修复	线上问题或高优先级修复
👷	CI/CD	流程、自动化、部署相关改动
🔒	安全	权限、鉴权、安全策略相关改动
⬆️	升级依赖	升级依赖版本
⬇️	降级依赖	回退或降级依赖版本


⸻

提交说明写法建议

提交描述应该聚焦“这次改了什么”，而不是写得过于宽泛。

推荐写法
	•	✨ 新增微信扫码登录
	•	🐛 修复用户信息为空报错
	•	♻️ 重构订单金额计算逻辑
	•	📝 更新接口返回字段说明
	•	🎨 调整会员页卡片布局
	•	🚀 优化商品列表查询性能

不推荐写法
	•	✨ 更新一下功能
	•	🐛 修复一些问题
	•	♻️ 优化代码
	•	📝 改文档
	•	🔧 改配置

问题在于信息量太低，后续回看提交历史时难以理解改动目的。

⸻

提交粒度建议

为了便于审查、回滚和定位问题，请尽量保持合理的提交粒度。

建议
	•	一个提交只做一类事情
	•	新功能与样式调整尽量分开提交
	•	重构与修复 bug 尽量分开提交
	•	大改动可以拆分为多个小提交
	•	提交标题只写最核心的改动点

示例

较好的拆分方式：

✨ 新增微信登录回调接口
🐛 修复登录状态未持久化问题
📝 更新微信登录流程文档

不推荐的方式：

✨ 新增微信登录并修复首页bug顺便改了文档和样式


⸻

禁止事项

请避免以下不规范提交方式：
	•	不要使用无意义描述作为完整提交信息
例如：update、fix、test、提交
	•	不要在一个提交中混入多个无关改动
	•	不要把提交标题写成过长的流水账
	•	不要为了“省事”频繁提交难以理解的内容
	•	不要使用与实际改动不符的表情

⸻

推荐工作方式

建议在提交前先自查以下内容：
	•	本次提交是否只包含一类核心改动
	•	提交信息是否带有表情
	•	提交描述是否足够明确
	•	是否包含无关文件改动
	•	是否符合项目已有命名和规范

⸻

最终规范

所有提交默认遵循以下格式：

<emoji> <简短描述>

请始终保证提交记录：
	•	清晰
	•	简洁
	•	可回溯
	•	易理解

良好的提交历史本身就是项目文档的一部分。
