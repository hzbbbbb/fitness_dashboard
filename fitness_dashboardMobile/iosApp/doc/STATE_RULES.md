# 状态管理约束

## 总原则

状态必须有唯一来源，且必须与其所属层级一致。

## 分层原则

- App 级状态放在共享层或明确的宿主集成层，不放在入口视图中临时拼接
- 页面级状态跟随页面所属实现层
- 临时 UI 状态可以留在视图层，但必须是纯展示状态
- 业务状态不能散落在多个 SwiftUI 视图局部变量中

## 当前项目约束

对于当前 iOS 容器工程，默认规则是：

- 业务状态优先由 `ComposeApp` 侧维护
- iOS 原生层只保留宿主所需的最小状态
- 宿主状态只应覆盖生命周期、权限、系统能力接入等平台问题

## 允许的宿主层状态

- 当前是否已完成容器挂载
- 系统权限状态
- 应用生命周期相关状态
- 与共享层桥接所必需的最小配置状态

## 不允许的宿主层状态

- 业务筛选条件
- 业务列表缓存
- 页面业务表单数据
- 仅为绕过共享层而新增的业务 ViewModel

## 修改规则

- 新增状态前必须先说明其归属层
- 如果一个状态既能在共享层管理，也能在宿主层管理，优先共享层
- 如果必须放在宿主层，必须在代码和文档中写清楚原因
