# UI 与页面开发约束

## 适用前提

当前 iOS 工程是宿主层，不是业务 UI 主战场。默认情况下，业务页面优先在共享层 `ComposeApp` 中完成，iOS 原生层只做最小容器包装。

## UI 开发基本原则

- 宿主层 UI 只负责入口、容器、必要平台能力
- 不在 iOS 宿主层重复实现共享层已有页面
- 没有明确需求时，不新增原生业务页面
- 能通过容器配置解决的问题，不要上升到页面重写

## 页面职责边界

`iosApp/iOSApp.swift` 的职责：

- 应用入口
- 场景挂载
- 不承载业务状态和业务逻辑

`iosApp/ContentView.swift` 的职责：

- 承载 `ComposeView`
- 提供最小宿主视图包装
- 不承担页面编排、数据拼装和业务判断

`ComposeView` 的职责：

- 建立 `UIKit` 与 `ComposeApp` 的桥接
- 不内嵌复杂平台业务逻辑

## 视觉改动约束

- 不因个人偏好重做整体视觉
- 不在没有设计输入的前提下擅自引入新视觉体系
- 宿主层如果出现原生占位页，风格必须简洁克制
- 任何临时页都必须标明其用途，避免伪装成正式业务页

## 页面开发禁止事项

- 禁止在宿主层堆叠多层导航来绕开共享层
- 禁止把共享层能做的页面直接挪到 SwiftUI 重写
- 禁止在入口页硬编码业务文案、业务路由或 mock 数据
- 禁止为了单次任务添加大而全的 UI 基础库
