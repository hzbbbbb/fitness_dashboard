# AI 协作与修改规范

## 修改前必须输出

AI 在开始改代码或文档前，必须明确输出以下内容：

1. 对任务目标的简要理解
2. 已阅读的 `doc` 文件清单
3. 计划修改的文件列表
4. 为什么这样改
5. 风险点与不确定项
6. 是否涉及超出当前阶段边界的内容

## 修改过程要求

- 优先最小改动
- 只解决当前任务直接相关的问题
- 不得顺手重构无关模块
- 不得用“未来可能需要”为理由提前加复杂抽象
- 优先复用现有结构和命名
- 保持宿主层职责清晰
- 修改时要考虑上游 `ComposeApp` 集成影响

## 修改后必须输出

AI 完成修改后，必须输出以下内容：

1. 修改了哪些文件
2. 每个文件改了什么
3. 实现了哪些结果
4. 哪些点没有做
5. 剩余风险是什么
6. 推荐的下一步是什么

## 文档与代码的一致性要求

- 如果代码边界发生变化，必须同步更新对应 `doc` 文件。
- 如果新增了规则，但没有明确执行入口，视为无效规则。
- 如果文档与代码冲突，AI 必须指出冲突点，不能假装一致。

## 代码风格要求

- 优先可读性
- 避免过度抽象
- 避免炫技写法
- 避免无必要的设计模式
- 保持组件职责清晰
- 页面、状态、数据模型、存储职责尽量分离

## 错误处理原则

如果遇到阻塞问题，AI 不得直接放弃，应至少输出：

1. 问题点
2. 最可能原因
3. 最小修复方案
4. 对现有功能的影响面

## 禁止事项

- 不要无提示覆盖大量代码
- 不要删除用户已有业务逻辑
- 不要私自修改 Git 历史
- 不要生成与当前任务无关的大量样板代码
- 不要把业务状态偷偷塞进 View 层或 App 入口
- 不要在 iOS 容器层复制一套共享层已有逻辑
